## 1. 项目总体规划

### 1.1 项目目标

**灵犀AI（超级员工）** 是一款基于 Electron/Tauri 的企业级 AI 社交媒体营销、获客和员工管理桌面应用。本次克隆重建项目的目标是：

1. **完整复现** 原项目的全部 12 个功能模块（AI创作、数字人、Sora2视频、AI拓客、AI个微、AI企微、AI人事、AI法务、企业智库、企业管理、数据罗盘、培训内容）
2. **技术升级**：将原项目从 Vue 2 + Element UI + Vuex + webpack 升级为 Vue 3 + Element Plus + Pinia + Vite + TypeScript
3. **架构优化**：使用 Tauri（首选）或 Electron 28+ 重构桌面外壳，提升性能和安全性
4. **完整闭环**：确保所有路由、组件、API 形成完整闭环
5. **自动化引擎**：完整实现 Playwright 多平台自动化（抖音、小红书、快手、Boss直聘）

### 1.2 技术选型

#### 前端技术栈

| 技术 | 版本 | 用途 | 选型理由 |
|------|------|------|----------|
| **Vue.js** | 3.x | 核心 UI 框架 | 成熟、生态好、Composition API |
| **Vite** | 5.x | 构建工具 | 极速 HMR、原生 ESM |
| **TypeScript** | 5.x | 类型安全 | 减少运行时错误，提高可维护性 |
| **Element Plus** | 2.x | UI 组件库 | 功能全面、文档完善 |
| **Pinia** | 2.x | 状态管理 | 替代 Vuex，更轻量、TypeScript 友好 |
| **Vue Router** | 4.x | 路由管理 | 官方路由，支持懒加载 |
| **ECharts** | 5.x | 数据可视化 | 功能强大、社区活跃 |
| **Axios** | 1.x | HTTP 请求 | 拦截器、请求取消、TypeScript 支持 |

#### 后端/桌面技术栈

| 技术 | 版本 | 用途 | 选型理由 |
|------|------|------|----------|
| **Tauri** | 2.0 | 桌面框架（首选） | 包体小（~10MB）、Rust 安全、性能好 |
| **Electron** | 28+ | 桌面框架（备选） | 与原架构相似、生态丰富 |
| **better-sqlite3** | - | SQLite 数据库 | 性能好、同步 API |
| **Playwright** | 1.x | 浏览器自动化 | 跨平台、多浏览器、API 完善 |
| **@nut-tree/nut-js** | 4.x | 桌面自动化 | 鼠标/键盘/屏幕控制 |
| **Express** | 4.x | 本地 HTTP 服务 | 成熟、轻量、中间件丰富 |

#### 工具链

| 工具 | 用途 |
|------|------|
| **Git** | 版本控制 |
| **pnpm** | 包管理（速度快、节省磁盘） |
| **ESLint + Prettier** | 代码规范 |
| **Husky + lint-staged** | Git 钩子 |
| **Vitest** | 单元测试 |
| **Playwright Test** | E2E 测试 |
| **Docker** | 本地服务容器化（可选） |

### 6.2 风险清单

| 序号 | 风险 | 等级 | 影响 | 应对方案 |
|------|------|------|------|----------|
| 1 | **远程 API 不可用或无文档** | 🔴 | 功能无法对接 | 准备 Mock API 方案，先开发前端界面和后端框架，API 就绪后接入 |
| 2 | **平台反爬升级** | 🔴 | 自动化失效 | 设计模块化自动化引擎，Cookie 和 UA 配置化，可快速切换策略 |

| 4 | **AI API 成本过高** | 🟡 | 功能受限 | 方案一：优化调用频率。方案二：接入多个 AI 供应商 |
| 5 | **Playwright 浏览器版本兼容** | 🟡 | 自动化不稳定 | 锁定 Chromium 版本，定期更新测试 |
| 6 | **数据迁移复杂** | 🟡 | 旧数据无法使用 | 编写数据迁移脚本，先解析旧版 JSON 格式 |
| 7 | **代码签名困难** | 🟡 | 安装包被拦截 | Windows: 购买代码签名证书。Mac: Apple Developer 账号 |
| 8 | **WebSocket/Protobuf 协议逆向难** | 🔴 | 微信协议对接受阻 | 使用官方 API 替代，或采用 HTTP 长轮询方案 |
| 9 | **Electron 安全警告** | 🟡 | 用户体验差 | 使用 Tauri 避免安全限制，如用 Electron 则优化安全配置 |
| 10 | **团队沟通效率低** | 🟡 | 开发效率下降 | 使用 GitHub Projects 管理任务，每日站会同步进度 |
| 11 | **性能瓶颈** | 🟡 | 响应慢 | 前端组件懒加载，数据库查询优化，渲染进程避免阻塞操作 |
| 12 | **版权/法律风险** | 🔴 | 法律纠纷 | 仅参考功能设计，不使用原项目代码，确保独立开发 |

### 6.3 应急计划

| 场景 | 触发条件 | 应急操作 |
|------|----------|----------|
| Phase 1 延期 | 超过 2 周未完成 | 简化 UI，使用 Element Plus 默认主题，减少自定义样式 |

| Phase 6 上架被拒 | 签名或审核问题 | 先提供免签名开发版，同步处理签名 |

### 7.2 目录命名规范

| 类型 | 命名规范 | 示例 |
|------|----------|------|
| 组件/页面 | PascalCase | `Sidebar.vue`, `MessageList.vue` |
| 路由文件 | camelCase | `wecom.ts`, `digitalHuman.ts` |
| Store | camelCase | `user.ts`, `platform.ts` |
| API 模块 | camelCase | `wecom.ts`, `digitalHuman.ts` |
| 类型定义 | PascalCase | `UserInfo.ts`, `ApiResponse.ts` |
| 工具函数 | camelCase | `format.ts`, `validate.ts` |
| 配置 | camelCase | `app.ts`, `api.ts` |
| 数据库迁移 | snake_case + 序号 | `001_create_customers.sql` |
| 测试文件 | 源文件同名 + `.spec` | `login.spec.ts` |

### 7.3 编码规范

// 1. 使用 TypeScript 严格模式
